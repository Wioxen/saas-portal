Configurar o **Cron Job** dentro do ambiente **Easypanel (Docker)** é o que transforma seu SaaS de um "projeto parado" em uma **máquina autônoma de milhões de acessos**. 

No Easypanel, você não usa o `crontab -e` tradicional do servidor Linux da mesma forma; você configura o agendamento **dentro do serviço da aplicação**. Isso garante que o script rode com as mesmas dependências e permissões do seu código PHP.

Aqui está a configuração de nível **Expert** para o seu império:

---

### 1. A Estrutura das Tarefas (Cron Logic)

Não coloque tudo em um único arquivo. Para escala extraordinária, divida em três frequências diferentes para não sobrecarregar a CPU da sua VPS:

| Tarefa | Frequência | Objetivo |
| :--- | :--- | :--- |
| **Scraper** | `*/5 * * * *` (5 min) | Detectar notícias urgentes e tendências no DOU e Google. |
| **Publisher** | `*/10 * * * *` (10 min) | Processar o que o Scraper achou, chamar a IA e postar no WP. |
| **Optimizer** | `0 3 * * *` (Todo dia às 3h) | Atualizar posts antigos, limpar banco e checar CTR. |

---

### 2. Configurando no Easypanel

No seu Dashboard do Easypanel, vá até o serviço da sua aplicação PHP e siga estas etapas:

1.  Vá na aba **"Resources"** ou **"Deploy"**.
2.  Procure pela seção **"Cron Jobs"**.
3.  Adicione as linhas seguindo o padrão:

**Comando para o Scraper:**
`php /var/www/html/sys/worker_scraper.php`
*(Agendamento: `*/5 * * * *`)*

**Comando para o Publisher:**
`php /var/www/html/sys/worker_publisher.php`
*(Agendamento: `*/10 * * * *`)*

---

### 3. O Script PHP "Worker" Robusto (Padrão de Elite)

O Claude deve escrever os scripts PHP de modo que eles sobrevivam a falhas. Use este modelo para evitar que o processo "trave" se a IA demorar a responder:

```php
<?php
// sys/worker_publisher.php

// 1. Impedir que o script rode duas vezes ao mesmo tempo
$lockFile = '/tmp/publisher.lock';
$lock = fopen($lockFile, 'c');
if (!flock($lock, LOCK_EX | LOCK_NB)) {
    exit("Processo já em execução. Aguardando próxima rodada.\n");
}

require_once 'vendor/autoload.php'; // Se usar Composer

// 2. Definir limites de tempo (IA pode demorar)
set_time_limit(300); // 5 minutos de fôlego

// 3. Lógica Principal
try {
    $db = connectMySQL(); // Sua conexão com a tabela 'trends_radar'
    
    // Busca o post com maior prioridade que não foi processado
    $trend = $db->query("SELECT * FROM trends_radar WHERE processado = 0 ORDER BY score_prioridade DESC LIMIT 1")->fetch();

    if ($trend) {
        echo "Processando: " . $trend['titulo_bruto'] . "\n";
        
        // EXECUTA A MÁGICA: IA -> IMAGEM -> POST WP -> INDEXING API
        $sucesso = executarFluxoExtraordinario($trend);

        if ($sucesso) {
            $db->query("UPDATE trends_radar SET processado = 1 WHERE id = " . $trend['id']);
            echo "✅ Postado com sucesso!";
        }
    }
} catch (Exception $e) {
    error_log("Erro no Worker: " . $e->getMessage());
}

// 4. Liberar o trava de execução
flock($lock, LOCK_UN);
fclose($lock);
```

---

### 4. A Ressalva de Ouro: "Ghost Mode" nas VPS

Como você tem **2 VPS**, o ideal é que os Cron Jobs de "Scraping" (que buscam dados externos) rodem em apenas **uma** delas, mas postem nos sites de **ambas**.

* **VPS 1 (Cérebro):** Roda os Cron Jobs de busca e processamento de IA.
* **VPS 2 (Escrava):** Apenas recebe os posts via API.

Isso evita que o Diário Oficial ou o Google vejam dois IPs diferentes "raspando" dados ao mesmo tempo, o que diminui o risco de bloqueio.

---

### 5. Monitoramento via Log

Dentro do Easypanel, você pode ver a aba **"Logs"**. Peça ao Claude para incluir muitos `echo` no código (como fiz acima). Assim, você abre o celular, olha o log do Easypanel e vê em tempo real:
* *"Buscando tendências..."*
* *"IA gerando conteúdo para o site Vagas..."*
* *"Indexação enviada ao Google!"*
